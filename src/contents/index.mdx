#### `Github`
- github issue와 PR을 이용한 코드 리뷰

#### `개발 디자인 파일 문서화`
- 요구사항이 추가되며 의사결정의 편의성을 위해 개발 히스토리를 기록 해야 했습니다. 현재 상황과 목표 상황을 비교해보는 AS IS - TO BE 양식을 사용해 탬플릿을 제작하고, 새로운 요구사항이 생길 때 마다 개발 과정을 기록하였습니다.

#### `코드 스타일 가이드 작성`
- 프로젝트가 진행함에 따라 팀원 각자 사용하던 스타일이 코드에 나타나는 문제가 있었습니다. 코드의 가독성을 높이기 위해 코드 스타일 가이드를 작성 했습니다.

#### `Eslint / Prettier / Husky`
- 팀 내에서 작성한 코드의 통일성을 위해 코드 포메터를 적극적으로 이용합니다. 또한 커밋 및 브랜치 네이밍 컨벤션을 맞추기 위해 Husky를 이용한 경험이 있습니다.

#### `Storybook`
- 디자이너와 원활한 소통을 위해 디자인 시스템을 만들고 스토리북으로 정리 해 본 경험이 있습니다. 이를 통해 공통 컴포넌트의 네이밍을 통일할 수 있었고 커뮤니케이션 비용을 줄일 수 있었습니다.
